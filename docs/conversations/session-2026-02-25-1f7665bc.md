> **Unified Agent State Layer — Implementation Plan**
> 
> **Context**
> 
> chatOS has four fragmented state systems: Chat SDK thread state in Redis, AI SDK messages in PostgreSQL, ephemeral client state in React, and the control metalayer in files. None of them share data — the bot is stateless between messages (no conversation history), the web app doesn't persist chat, and there's no cross-platform session continuity.
> 
> The goal: unify all agent state under a single filesystem-shaped abstraction where `MessagePart[]` is the canonical data format, sessions are shared across platforms (web + Slack), and the control metalayer extends to observe runtime behavior. "The agent IS the app — files are all you need."
> 
> The codebase is in skeleton phase (no persistence wired in either app), making this the ideal time.
> 
> **Architecture**
> 
> ```
> StorageBackend (interface: read/write/list/delete/exists)
>   ├── LocalStorageBackend   (.agent/ dir — bot, dev)
>   ├── VercelBlobBackend     (Vercel Blob — web, prod)
>   └── MemoryBackend         (tests)
>        │
>        ▼
> AgentStateStore (sessions, messages, memory, observations)
>        │
>   ┌────┴────┐
>   │         │
>   ▼         ▼
> AgentStateAdapter          Web API routes
> (Chat SDK StateAdapter     (read/write via store)
>  bridge — Redis for
>  locks/dedup, store
>  for durable state)
>   │         │
>   ▼         ▼
> Bot (Elysia)    Web (Next.js)
>   │                │
>   └── same sessions ─┘
> ```
> 
> PostgreSQL remains for user/auth and as a query-optimized projection (chat list sidebar, search). Redis remains for ephemeral coordination (locks, dedup, subscriptions). The filesystem store is the canonical source of truth for agent state.
>
